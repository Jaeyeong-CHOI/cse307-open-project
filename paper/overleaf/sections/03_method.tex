\section{Method}
\subsection{Language Setup}
We define a Python-like confusion language by replacing selected Python tokens with aliases. The transformation enforces:\begin{enumerate}
  \item one-to-one mapping between each source token and its alias;
  \item no alias sharing among control keywords (e.g., \texttt{if} and \texttt{elif});
  \item token-boundary replacement while preserving comments and string/regex literals.
\end{enumerate}

\subsection{System Pipeline}
The implementation combines OCaml transformations with Python evaluators:
\begin{itemize}
  \item alias/lineage validation and roundtrip conversion in \texttt{ocaml-confusion-lang};
  \item generation summary and mapping scripts in \texttt{ocaml-confusion-lang/scripts};
  \item metric aggregation and schema validation via
  \texttt{batch\_report\_summary.py}, \texttt{generate\_metric\_snapshot.py},
  \texttt{validate\_summary\_payload.py}, and \texttt{validate\_metric\_schema.py}.
\end{itemize}

\subsection{Metrics}
We define:
\begin{itemize}
  \item ACR (Alias Compliance Rate)
  \item PRR (Python Reversion Rate)
  \item ESR (Execution Success Rate)
  \item MFB (Manual Fix Burden)
  \item LGP (Learning Gain Proxy)
\end{itemize}

\subsection{Experiment Configuration}
Primary runs are controlled by fixture task/task-alias pairs (e.g., \texttt{fixture-task-set-v1}, \texttt{fixture-alias-v1}) and prompt conditions (\texttt{strict} baseline).
Live API calls are executed through the batch path when credentials are available; otherwise, fixture replay is used for reproducibility checks.
