\section{Method}
\subsection{Research Question and Hypotheses}
We study whether code LLMs obey transformed Python-like syntax under explicit alias constraints.

\textbf{RQ1}: Under strict alias instructions, how often do models retain transformed tokens vs. revert to canonical Python keywords?

\textbf{RQ2}: Are failures dominated by infrastructure issues or by behavioral non-compliance after infrastructure stabilization?

\textbf{H1}: Even with valid API execution, strict alias compliance remains low because models revert to dominant Python priors.

\subsection{Language Setup and Constraints}
We define a confusion language by replacing selected Python tokens with aliases. Constraints:
\begin{enumerate}
  \item one-to-one mapping between each source token and alias;
  \item no alias sharing among control keywords (e.g., \texttt{if}/\texttt{elif});
  \item token-boundary replacement;
  \item comment/string literal protection.
\end{enumerate}

\subsection{Pipeline Architecture}
The evaluation stack combines OCaml transformation and Python orchestration:
\begin{itemize}
  \item \textbf{Transform/validate layer}: alias checks, collision prevention, roundtrip normalization;
  \item \textbf{Batch execution layer}: prompt-version evaluation and result serialization;
  \item \textbf{Schema gate layer}: summary payload and metric schema validation.
\end{itemize}

Key scripts include \texttt{run\_gpt54\_eval.py}, \texttt{batch\_report\_summary.py},
\texttt{generate\_metric\_snapshot.py}, \texttt{validate\_summary\_payload.py}, and \texttt{validate\_metric\_schema.py}.

\subsection{Metrics and Judge Protocol}
We use two metric families:
\begin{itemize}
  \item \textbf{Fixture aggregate metrics}: ACR, PRR, ESR, MFB, LGP.
  \item \textbf{Prompt-batch judge outcomes}: pass/fail with violation taxonomy.
\end{itemize}

Judge protocol checks:
\begin{enumerate}
  \item strict alias presence and original keyword leakage;
  \item alias-to-python normalization validity (AST parseability);
  \item target skeleton consistency for the benchmark task pattern.
\end{enumerate}

\subsection{Threats to Validity}
\textbf{Internal validity}: false negatives may occur if judge rules are stricter than practical human acceptance.

\textbf{Construct validity}: pass/fail under one task skeleton does not represent full software engineering competence.

\textbf{External validity}: current runs focus on a constrained prompt-family and may not generalize across broader task distributions.
