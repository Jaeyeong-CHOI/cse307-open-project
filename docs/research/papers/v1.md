## 1. 연구 배경과 문제 정의

최근 LLM의 코딩 성능이 빠르게 향상되면서, 많은 모델이 Python 문법을 매우 강하게 우선 생성하는 경향을 보입니다.
이 연구는 다음 질문을 다룹니다.

> “Python과 유사하지만 일부 문법 표현을 변형한 언어를 제시했을 때,
> 모델이 변형 규칙을 따르는가, 아니면 익숙한 Python 문법으로 회귀하는가?”

또한 교육적 관점에서,
생성 코드를 그대로 신뢰하지 않고 검수·수정하는 과정이 학습 효과를 높일 수 있는지도 함께 탐색합니다.

---

## 2. 접근 방법 (v1)

v1에서는 본격 대규모 평가 전, 연구 파이프라인의 기초 신뢰성을 먼저 확보했습니다.

### 2.1 문법 설정
- Python 키워드(`in`, `def`, `for`, `return`, `if`, `elif`)에 alias를 부여
- 충돌 방지를 위해 키워드별 alias 1:1 매핑 강제
- `if`/`elif` 동일 문구 사용 금지 규칙 적용

### 2.2 도구체인 (OCaml)
`ocaml-confusion-lang` CLI를 통해 아래 기능을 구현했습니다.

1. `validate`: alias 세트 검증
   - 필수 키 존재 확인
   - 중복 alias 감지
2. `transform`: Python -> 혼동 언어 치환
   - 단어 경계 기반 치환
   - 주석 구간 보호
3. `roundtrip`: Python -> alias -> Python 복원 일치성 점검
   - mismatch 발생 시 첫 diff 라인 출력

### 2.3 문서/실험 구조화
- 연구 로드맵: `docs/research/roadmap.md`
- 연구 로그: `docs/research/log/`
- 시연 프리셋: `docs/presets/final-demo-aliases.md`
- 과거 기록: `docs/archive/`

## 3. v1 결과 요약

### 3.1 확보된 성과
- 연구 저장소 구조를 “활성 연구/시연 프리셋/아카이브”로 분리하여 가독성 개선
- alias 충돌(특히 `if`/`elif`)을 사전 검증 가능한 체계 구축
- roundtrip 경로를 통해 변환 파이프라인의 기본 일관성 점검 기반 확보

### 3.2 현재 관측 (정성)
- “같은 문구를 여러 키워드에 배정”하면 역치환 모호성이 크게 증가함
- 키 길이/부분 문자열 관계가 치환 안정성에 영향을 줌
- 사용자/시연 관점에서 “명시적이고 짧은 alias”가 오류 추적에 유리함

## 4. 비판적 검토 (한계)

v1은 파이프라인 구축 단계이므로, 아래 한계가 있습니다.

1. 정량 실험 미완료
   - 모델별 ACR/PRR/ESR 통계치가 아직 충분히 쌓이지 않음
2. 실행 환경 의존성
   - 로컬에서 `dune` 미설치 시 즉시 재현이 제한됨
3. 파서 정교화 필요
   - multiline/triple-quote/복합 경계 케이스를 더 강건하게 다뤄야 함
4. 교육 실험 데이터 부재
   - 학생군 전/후 비교 데이터는 v1 범위를 벗어남

---

## 5. 다음 버전(v2) 계획

v2에서는 “정량 결과”를 본격적으로 확보합니다.

1. 벤치마크 태스크셋 v1 확정
2. 모델별 실험 배치 실행(비용 통제 포함)
3. 지표 산출
   - ACR(규칙 준수율)
   - PRR(Python 회귀율)
   - ESR(실행 성공률)
4. 실패 taxonomy 정리
   - 충돌/누락/부분치환/연쇄치환/포맷 붕괴
5. 결과 표/그래프를 `docs/research/results/`에 버전별 저장
