# [v2] Python 혼동 유도 언어 연구 결과 논문형 초안

- 버전: v2.0 (논문 통합 초안)
- 작성일: 2026-03-23
- 상태: 방법론·코드 변화·실험 결과를 통합 정리한 누적판(인프라 제약으로 부분 실측)

## 초록
본 연구는 Python과 유사하지만 일부 구문을 의도적으로 치환한 \"혼동 유도 언어(confusion-inducing Python-like language)\"에서, 코드 생성 모델의 문법 준수도와 Python으로의 회귀 경향을 정량 분석한다. 연구의 핵심은 모델 성능 자체의 절대평가가 아니라, 지시된 변형 문법을 따르지 못하고 기존 Python 관성에 회귀하는 패턴을 측정 가능한 방식으로 재현·해석하는 것이다.

우리는 OCaml 기반 변환·검증 파이프라인(치환·라운드트립·요약/메트릭 검증)을 먼저 안정화한 뒤, fixture 기반 배치와 제한된 LLM 호출 흐름을 통해 ACR/PRR/ESR 지표를 산출했다. 또한 선행연구 맥락에서 AI 보조 개발의 교육적 영향(생산성 vs 검증 능력), 실험 제어(중복 alias·파이프라인 drift) 통제를 함께 반영하였다.

## 1. 연구 배경 및 동기
1. 모델이 Python 코드를 강하게 선호한다는 점은 공학 실무에서도 반복 확인되며, 실제 실험은 이러한 관성이 형식적 제약 조건 아래에서 어떻게 무너지는지 보여주어야 한다.
2. 본 연구는 \"문법 준수\"와 \"회귀 유혹\"를 분리 측정하기 위해, 변형된 Python-like 언어를 실험 자극으로 사용한다.
3. 교육적으로는 자동 생성 의존이 아닌 검증/수정 역량을 강화하기 위해, 단순 정답률이 아닌 수동 수정 부하와 재학습 효과 지표를 병행한다.

## 2. Related Work (선행연구 확장)

### 2.1 코드 생성 대가설과 벤치마크 기초
- **코드 생성 성능의 전초점**: Codex/BPE/ChatGPT 계열 기반의 대형 코드 모델은 자연어-코드 대응에서 눈에 띄는 성능 향상을 보였으나, 여전히 구문/의도 제약이 높은 환경에서 불안정성이 크다. 대표적으로 Codex 평가 프레임워크(HumanEval) 및 MBPP 계열 연구가 짧은 함수 합성 중심의 정량 지표를 제시한다. \(Chen et al., 2021; Austin et al., 2021\)
- **임베딩/사전학습 기반 코드 모델 발전**: CodeBERT, CodeT5, UniXcoder 계열은 코드 표현 학습을 강화해 코드 이해·생성 성능을 끌어올렸으나, 본 연구의 핵심인 `transform 규칙 준수`(alias collision 및 준문법 정합) 보장 문제는 아직 별도 축으로 다루어지는 경우가 적다.

### 2.2 제약/정렬/리스크 라인
- **명령 준수 및 정렬**: Instruction-tuning 및 RLHF 기반 접근은 자연어 지시 순응을 높였지만, 지시가 언어 규격 자체(토큰 규칙, keyword 집합, alias 동치성)로 바뀔 때의 붕괴 특성은 덜 다뤄졌다. \(Ouyang et al., 2022; Brown et al., 2020\)
- **실무 소프트웨어 과제의 난이도**: SWE-bench는 실제 GitHub issue 기반 패치 문제를 통해 짧은 함수 합성 벤치마크의 한계를 드러낸다. 즉, 고성능 모델이라도 구조적 제약/긴 컨텍스트·다파일 편집에서 취약함이 반복 확인된다. \(Jimenez et al., 2023\)
- **제약 기반 생성의 확장 가능성**: 타입 제약/구문 제약 디코딩은 최근 강화되는 연구 흐름으로, 본 연구의 혼동 언어 제약은 문맥 기반 제약 강화의 실증적 장으로 볼 수 있다. \(Mündler et al., 2025\)

### 2.3 학습효과/디버깅·검수 관점
- 연구 초기 단계의 내부/외부 문헌은 AI 보조 사용이 속도 이점을 주는 동시에, 초보자의 코드 검증 능력·개념 정착에 제약을 줄 수 있음을 시사한다.
- 본 연구는 이에 대응해 정확도 지표 대신 **수동 수정 부하(MFB), 교육 성장 예비 지표(LGP), reversion rate**를 함께 본다.

현재 본 절의 핵심 문헌은 아래 항목을 통해 v2 정리에 통합되었고, 추가 인용은 문헌 검증 로그 기준으로 확장 예정:
- Chen et al., 2021 — *Evaluating Large Language Models Trained on Code*
- Austin et al., 2021 — *Program Synthesis with Large Language Models*
- Feng et al., 2020 — *CodeBERT: A Pre-Trained Model for Programming and Natural Languages*

## 3. 시스템 구성

### 3.1 언어/표기 체계
- 대상 토큰: `in`, `for`, `if`, `elif`, `def`, `return`, `while`, `print` 등
- 원칙:
  1) 원형 Python token별 1:1 alias 매핑
  2) `if`/`elif` 동시 동일 alias 금지
  3) 단어 경계 기반 치환
  4) 문자열/주석/리터럴 보호

### 3.2 파이프라인
- 유효성: `validate`/alias 충돌 검사/스키마 검증
- 변환: `transform`/`roundtrip`
- 실험/리포트: `batch_report_summary.py`, `generate_metric_snapshot.py`, `validate_*` 계열
- 품질: 결과 항목의 lineage, provenance, schema 일치성, 중복 run-라인 보호

### 3.3 실험 설계 제어
- task-set: `fixture-task-set-v1`, `fixture-alias-v1` 중심
- 실험 조건: strict 모드 포함
- 반복성: summary/json/csv 산출 + validate 스크립트 일괄 실행
- 지표:
  - **ACR** (Alias Compliance Rate)
  - **PRR** (Python Reversion Rate)
  - **ESR** (Execution Success Rate)
  - **MFB** (Manual Fix Burden)
  - **LGP** (Learning Gain Proxy)

## 5. 실험 결과

### 5.1 실측 배치 제약 (업데이트)
- 초기에는 인증(401)으로 차단되었으나, 2026-03-23 저녁 재실행에서 API 호출 경로가 안정화되어 `v1~v120` 단일 배치 실측 JSON을 다층 조건으로 누적 확보했다.
  - `prompt-batch-v1-120.gpt4o-mini.2026-03-23.json` (`total=120`, `passed=0`, `failed=120`)
  - `prompt-batch-v1-120.gpt41-mini.2026-03-23.json` (`total=120`, `passed=0`, `failed=120`)
  - `prompt-batch-v1-120.gpt41-nano.2026-03-23.json` (`total=120`, `passed=0`, `failed=120`)
  - `prompt-batch-v1-120.gpt54-mini.2026-03-23.json` (`total=120`, `passed=0`, `failed=120`)
  - `prompt-batch-v1-120.gpt4o.2026-03-23.json` (`total=120`, `passed=2`, `failed=118`)
- 다섯 배치 모두 HTTP 실패는 `0`건이며, 실패 원인은 HTTP 레벨이 아니라 alias 준수/구문 위반(judge violation) 중심으로 관측된다.
- `gpt-4o` 배치에서 full-range 기준 최초 pass 2건이 확인되었고, 이는 기존 전모델 `pass=0` 구간에서 작은 양의 신호로 해석된다.

- `gpt-4o-mini` contextpack 누적 확장(v21~v120, 2026-03-24): `total=100`, `passed=0`, `failed=100`, `http_failures=0`을 기록. legacy 구간(v21~v120) 대비 `delta_avg_score=+1.100`, `delta_passed=+0`, `delta_nonzero_score=+0`.
- 동일 조건(`gpt-4o`, contextpack, v1~v120) 재실험(`2026-03-24`)에서 `passed=3`, `failed=117`로 pass가 1건 증가했으며 `delta_avg_score=+5.750`, `delta_nonzero_score=+5`를 기록했다.
- 추가로 `2026-03-24` 동일 날 `gpt-4o-mini` contextpack 조건으로 `v1~v20` 부분 재실행을 수행하여 `passed=0`, `failed=20`, `http_failures=0`을 확인했고, 해당 구간 비교에서 `delta_avg_score=0`, `delta_passed=0`, `delta_nonzero_score=0`으로 재현성은 양호하나 성능 개선은 관측되지 않았다.
- 추가 통제실험으로 `gpt-4o legacy(no contextpack)` full-range를 동일 구간(v1~v120)에서 실행한 결과 `passed=1`로 관측되었고, contextpack ON 대비 `delta_passed=+1`, `delta_avg_score=+0.583` 개선이 확인되었다.
- 후속 통제실험으로 `gpt-4o-mini legacy(no contextpack)` full-range(v1~v120)를 별도 재실행한 결과 `passed=0`으로 유지되었으나, contextpack ON 대비 `delta_avg_score=+1.083`, `delta_nonzero_score=+3` 개선이 확인되었다.
- 추가 통제실험으로 `gpt-5.4-mini legacy(no contextpack)` full-range(v1~v120)를 실행한 결과, `passed=0` 동률이지만 **legacy가 contextpack 대비** `delta_avg_score=+2.500`, `delta_nonzero_score=+3`로 우세했다.
- 추가 통제실험으로 `gpt-5.4-mini` contextpack(full-range v1~v120)를 실행해 `passed=3`, `failed=117`를 기록했고, legacy 대비 `delta_avg_score=+12.583`, `delta_nonzero_score=+13`, `delta_passed=+3`으로 contextpack 유의 개선을 확인했다.
- 추가 통제실험으로 `gpt-4.1-mini` contextpack(full-range v1~v120)을 2026-03-24에 재실행해 `passed=1`, `failed=119`, `http_failures=0`을 기록했다.
- legacy 대비 비교에서 `delta_avg_score=-2.250`, `delta_nonzero_score=-12`, `delta_passed=+1`으로, contextpack이 오히려 legacy 대비 점수 밀도를 낮추는 방향이었다.
- `python parse failed after normalization` 빈도는 gpt-5.4-mini(95)와 gpt-4.1-mini(96)가 낮은 편이었고, gpt-4o-mini(109), gpt-4.1-nano(111)가 상대적으로 높았다(신규 gpt-4o는 `original keyword used` 편중).
- `gpt-5.3-codex`는 모델 가용성 probe 1건에서 HTTP 404로 종료되어 현재 평가 엔드포인트 비지원으로 분류했다.
- gpt-4.1-nano 배치에서 네트워크 read timeout 2건이 있었으나 인증/런타임 전면 실패로 이어지지는 않았다.
- 즉, 병목은 인프라 인증 이슈에서 **프롬프트/규칙 설계 난이도**로 명확히 이동했다 (당일 full-range+변형조건 누적이 누적 확장됨).

## 6. 코드 기여
- alias 충돌 방지 정책을 테스트 가능한 함수 레이어로 정리

## 7. 토의
기술 레이어는 안정, 평가 레이어는 제한됨:
1. Fixture 기반 검증·정합성은 성숙.
2. 실측 호출은 복구되었고, 현재 제한은 인증이 아니라 규칙 준수 실패율(0 pass) 자체.
3. 따라서 다음 단계는 대규모 모델 비교와 함께 prompt redesign/constraint decoding 계열 개선 실험이 필요.

## 8. 결론 및 향후 계획
- 핵심 기여: Python-like 혼동 언어 준수/회귀를 재현 가능한 단위로 측정할 수 있는 실행 기반 완성.
- 다음 단계:

  2) 모델군/프롬프트 조건별 ACR/PRR 비교 + top violation taxonomy 자동 요약
  3) mismatch taxonomy와 학습개입 지표를 함께 수치화
