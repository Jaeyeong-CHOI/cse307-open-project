# [v2] Python 혼동 유도 언어 연구 결과 논문형 초안

- 버전: v2.0 (논문 통합 초안)
- 작성일: 2026-03-23
- 상태: 방법론·코드 변화·실험 결과를 통합 정리한 누적판(인프라 제약으로 부분 실측 구간 존재)

## 초록
본 연구는 Python과 유사하지만 일부 구문을 의도적으로 치환한 "혼동 유도 언어(confusion-inducing Python-like language)"에서, 코드 생성 모델의 문법 준수도와 Python으로의 회귀 경향을 정량 분석한다. 연구의 핵심은 모델 성능 자체의 절대평가가 아니라, 지시된 변형 문법을 따르지 못하고 기존 Python 관성에 회귀하는 패턴을 재현 가능하게 측정하는 것이다.

우리는 OCaml 기반 변환·검증 파이프라인(치환·라운드트립·결과 요약·스키마 검증)을 먼저 안정화한 뒤, fixture 기반 배치와 제한된 LLM 호출 흐름을 통해 ACR/PRR/ESR 지표를 산출하였다. 또한 선행연구 맥락에서 AI 보조 개발의 교육적 영향(생산성 대비 이해도 저하 가능성), 실험 제어(중복 alias·파이프라인 drift) 통제를 함께 반영하였다.

## 1. 연구 배경 및 동기
1. 모델이 Python 코드를 강하게 선호한다는 점은 널리 보고되었고, 실제 현업에서는 사용자가 다른 문법 규약을 도입하더라도 기존 관성으로 회귀할 위험이 있다.
2. 본 연구는 "문법 규칙 준수"와 "회귀 유혹"를 분리 측정하기 위해, 변형된 Python-like 언어를 실험 자극으로 사용한다.
3. 또한 혼동 언어 실험은 단순 정확도 비교가 아닌, **교육적으로 검수·수정(Manual Review) 과정의 필요성**을 정량화하기 위한 scaffolding으로도 유효하다.

## 2. 관련연구 (선정)
### 2.1 교육·개발 생산성-이해도 트레이드오프
- 선행 검토 항목: AI 코딩 보조 사용이 주니어 개발자의 디버깅 성능/이해도에 미치는 영향에 대한 1차 검토 (상태: 1차)
- 시사점: 속도 향상이 곧바로 품질 이해력 향상으로 이어지지 않을 수 있어, 본 연구의 '회귀율/수동 수정 부하' 측정 동기가 정당화됨.
- 제약: 아직 공개 원문 검증은 보강 단계(참고문헌 상세화 필요)

### 2.2 규칙 충돌·구문 일관성 통제
- 내부 실험/로그에서 `if/elif` 동시 alias와 같은 충돌이 역치환 모호성과 오류 해석 왜곡을 크게 유발함을 확인.
- 본 논문에서는 이를 통제 변수로 다루고, 충돌 유발 preset를 분리 배제하거나 조기 실패로 처리하도록 설계.

## 3. 시스템 구성

### 3.1 언어/표기 체계
- 대상 키워드: `in`, `for`, `if`, `elif`, `def`, `return`, `while`, `print` 등
- 공통 원칙:
  1) 원형 Python token별로 **1:1 alias mapping** 강제
  2) `if`/`elif` 동시 동일 alias 금지
  3) word boundary 중심 치환
  4) 주석/문자열 구간 보호

### 3.2 파이프라인
- 유효성 검증: `ocaml-confusion-lang/scripts/validate_task_set.py` 계열 검증, alias 충돌 및 schema 검사
- 코드 변환: `transform`/`roundtrip`
- 배치 실험: `run_gpt54_eval.py`(호출 경로), `batch_report_summary.py`, `generate_metric_snapshot.py`
- 스키마 검증: `validate_summary_payload.py`, `validate_metric_schema.py`
- 품질 제어: 실패 항목 분리, task-set/task-id lineage 추적, provenance 필드 확장

### 3.3 실험 설계 제어
- task-set 기준: `fixture-task-set-v1`, `alias`는 `fixture-alias-v1`
- 프롬프트 조건: strict 기준 포함
- 반복성: 결과물(요약 JSON/CSV/Markdown) + 메트릭 스냅샷 저장 + 검증 스크립트 기반 일관성 체크
- 지표 정의:
  - **ACR**: Alias Compliance Rate
  - **PRR**: Python Reversion Rate
  - **ESR**: Execution Success Rate
  - **MFB**: Manual Fix Burden
  - **LGP**: Learning Gain Proxy

## 6. 코드 수준 기여 정리
- alias 충돌 방지 정책을 테스트 가능한 형태로 제도화
- 결과 소비자의 파싱 부하를 줄이기 위해 다중 출력 포맷(compact JSON/JSONL/TSV/markdown) 구현
- 출력의 기계 판독성 향상(프로브 기반 provenance)
- 문서·로그 운영 체계를 단계별(상태/로그/digest/누적 결과)로 정리

## 7. 논의
현재 결과는 다음 두 층위로 해석한다.
1. **기술층**: pipeline/validation은 정상 동작하며 fixture 기준 재현 실험은 가능
2. **평가층**: 실측 LLM 호출 제약(인증/실행 환경)이 있어 PRR/ESR의 일반화된 모델 비교는 미완

이는 모델 성능 결론을 과도하게 일반화하지 못하다는 강한 한계를 의미한다. 대신, 연구가 보여준 핵심은 **실험 인프라의 재현성/검증성/오류 분해가 확보된 상태**라는 점이다.

## 8. 결론 및 향후 계획
- 본 연구는 혼동 유도 언어 환경에서 LLM 준수도와 회귀율을 측정하기 위한 실험 기반을 완성했다.
- 다음 단계:
  1) 유효 LLM 엔드포인트 확보 후 1차 fixture 확장 배치 재실행

  3) task-set별 누적곡선(ACR/PRR/ESR) 및 mismatch taxonomy 분해 보고
  4) 학생군 검수 학습효과(수동 수정 부하, 이해도 전후 평가) 통합
